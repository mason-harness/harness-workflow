# 维护 `CLAUDE.md` 与根目录知识库

## 核心原则

**先定位一个目标项目，再用已验证事实更新一个 `CLAUDE.md` 和一套根目录 `repository/`；宁可少写，也不要把猜测、临时需求或多项目混合信息写成规则或知识。**

维护单个项目的工作文档体系：
- `CLAUDE.md` 记录 Claude 在本项目如何工作：执行边界、常用命令、开发规则、危险操作、完成前验证。
- `repository/` 记录稳定项目知识：术语、架构、模块边界、集成约束、长期设计决策。

## 使用时机

- 初始化、更新、重构或本地化 `CLAUDE.md`
- 初始化、补全或重组根目录 `repository/`
- 检查工作文档是否过时或职责混乱
- 从 README、Wiki、OpenSpec 或团队规范同步长期工作说明与稳定项目知识
- 补充命令、架构边界、模块说明、外部集成约束

**不用于**：解释概念、记录一次性需求、批量维护多项目文档、把运行中任务或今日计划写进长期知识。

## 关键规则

1. **必须先读后写** - 读取现有 `CLAUDE.md`、`repository/`（如存在）和能证明结论的项目文件；不凭框架常识写事实
2. **必须只选一个目标项目** - 多应用/包/monorepo 时，先让用户指定项目目录
3. **必须区分文档职责** - `CLAUDE.md` 写执行规则；`repository/` 写稳定知识；README 写人类上手；OpenSpec 写 artifact 规则与项目流程
4. **必须可执行、可维护** - 规则具体到能执行和检查；知识具体到能复用和定位；删除口号、死路径、错误命令
5. **必须把 `repository/` 视为根目录标准接口** - 对外暴露固定目录结构，避免把稳定知识散落在临时文档中

## Hardness 模型

术语：
- **MUST / MUST NOT**: 硬约束，不因速度或催促绕过
- **SHOULD**: 默认行为，可被更强证据覆盖
- **MAY**: 可选增强
- **STOP**: 门禁条件；暂停、报告冲突/缺证据/待决策，不继续猜测

**实现机制认知**：`CLAUDE.md` 通过会话启动注入上下文实现软约束。要转化为硬门禁需配合 **Claude Code Hooks（PreToolUse）**。

## 文档职责模型

| 载体 | 应承载内容 | 不应承载内容 |
|---|---|---|
| `CLAUDE.md` | 命令、执行边界、Do/Don't、Before Finishing、Dangerous Operations | 术语词典、模块百科、一次性需求、具体 harness skill / slash command 使用步骤 |
| `repository/` | 术语、架构、模块职责、集成约束、长期决策、Harness/OpenSpec 使用流程知识 | 今天的任务、流程口号、未验证命令、具体会话上下文 |
| README | 人类上手说明 | Claude 专用执行规则 |
| OpenSpec | artifact 规则、Hardness、变更流程 | 详细编码手册、命令大全 |

如果目标项目需要长期保存 Harness/OpenSpec 的使用流程，应放入 `repository/harness-workflow.md` 或现有知识库入口；`CLAUDE.md` 只保留目标项目长期执行规则，不写“调用某个 skill”的操作步骤。

## 工作流程

### 1. 定位目标和语言

- 默认目标是当前工作目录的单个项目
- 如果仓库包含多个候选项目（`apps/*`、`packages/*`、多个 `package.json`），且用户未指定目标，**STOP 并询问**要更新哪个项目
- 只有用户明确要求“仓库根目录 / monorepo-wide”或现有根 `CLAUDE.md` 明确承担全仓职责时，才更新根级文档
- 确认输出语言：沿用现有语言；新建时匹配用户语言

### 2. 读取最小事实来源

至少读取：
- 目标 `CLAUDE.md`（如存在）
- 根目录 `repository/`（如存在）
- 包管理、脚本、依赖、构建、测试、lint 相关文件
- 用户提供的团队规范

按技术栈读取关键文件：入口、路由/模块、服务层、配置、测试配置。

### 3. 验证再写入

详见 `references/evidence-guidelines.md`。

**关键**：
- 命令必须来自 `package.json` scripts、Makefile 或任务配置。
- 稳定项目知识必须来自代码结构、配置、稳定规范或已有知识库证据。
- 不保留无可执行来源支持的旧命令，不把未验证假设写成知识。
- 用户期望不存在命令有效时 STOP。

### 4. 增量更新模式

刷新/更新请求时，默认不从头重写。

分类现有内容：
- **保留**：准确且有证据支持
- **更新**：部分过时或措辞需改进
- **移动**：内容正确但放错载体（例如从 `CLAUDE.md` 移到 `repository/`）
- **删除**：过时、被证据否定或超出范围
- **未知**：无法验证

优先局部章节编辑。仅当文档矛盾、不可用或用户明确要求时才完全重组。

### 5. 更新文件

- 优先增量编辑；结构混乱时可重组，但保留准确内容
- 删除过时、重复、不可验证、一次性或职责冲突的内容
- 把团队规则转成行动指令；不整段粘贴
- 大范围重写前展示计划并确认；小修正可直接改
- 初始化时同时建立 `CLAUDE.md` 与根目录 `repository/` 的职责边界

结构模板和建议目录详见 `references/structure-templates.md`。

## 红旗警告 - STOP

详见 `references/common-mistakes.md` 的“红旗警告”章节。

**关键 STOP 条件**：
- 多项目不明确
- 用户命令与项目不一致
- 业务需求伪装成长期知识
- 未经验证的常识
- 把稳定知识继续堆在 `CLAUDE.md` 或其他单文件中
- 复制其他文档

## 技能交接

以下交接规则只用于当前协作流程，不得原样写入目标项目的 `CLAUDE.md`。

### 交接给 Hook 配置流程

生成或更新 Dangerous Operations 章节时，只在目标文档中保留概念性交接：高风险操作可升级为 Hook 硬门禁，以实现模型无法绕过的拦截；不要写具体 skill 名称、slash command 或一次性操作步骤。

高风险操作（数据删除、生产部署、配置变更）时，向用户说明风险与 Hook 硬门禁价值；如需记录长期流程，将 Harness/OpenSpec 使用流程写入目标项目 `repository/`，不要写进 `CLAUDE.md`。

### 交接给 OpenSpec 配置流程

当编码/测试/流程规则需要投射为 OpenSpec artifact 规则，或需要让 OpenSpec 读取根目录知识库作为项目上下文时，只把稳定流程知识放入 `repository/`；不要把具体 skill 使用方式写入 `CLAUDE.md`。

### 交接给 issue/spec docs

一次性功能需求。

### 交接给用户

目标歧义、缺失命令、未解决的来源冲突。

## 质量检查清单

完成前检查：

- [ ] 已读取目标 `CLAUDE.md`、现有 `repository/` 和必要项目配置
- [ ] 已确认只有一个目标项目；多项目歧义已向用户确认
- [ ] 命令与实际 scripts/任务配置一致，缺失命令未写成事实
- [ ] 技术栈、架构、路径和规则有代码/配置或稳定规范依据
- [ ] 已把稳定项目知识放入 `repository/`，没有继续塞进 `CLAUDE.md`
- [ ] 已删除过时内容、死路径、错误命令和重复段落
- [ ] 未写入一次性业务需求、当天 TODO、聊天上下文或提交历史
- [ ] Do / Don't / Before Finishing 可执行、可验证
- [ ] **Before Finishing 章节存在且包含可执行检查清单**
- [ ] **Dangerous Operations 章节存在且标注 STOP 条件**
- [ ] **禁止的自我合理化章节存在（推荐）**
- [ ] 未记录凭据、账号信息或私有访问地址
- [ ] 未与 README / OpenSpec / repository 重复维护大段内容
- [ ] 证据映射已完成（来源已验证）
- [ ] 命令已检查存在性
- [ ] 冲突已按优先级解决或已报告用户
- [ ] 增量刷新保留了准确的现有内容
- [ ] OpenSpec 交接已识别且不混合职责

## 输出要求

向用户汇报：

1. 修改了哪个 `CLAUDE.md` 与哪些知识库文件
2. 新增、删除或重组了哪些内容
3. 哪些内容来自代码/配置验证，哪些来自用户稳定规则
4. 哪些内容被移入 `repository/` 以及原因
5. 哪些请求被重定向到更合适的文档及原因
6. 如有未确认内容，列出待确认项
7. 如只生成建议未写文件，明确说明
8. **如生成或更新 Dangerous Operations 章节，提示配置 Hook**
