# 常见错误

| 错误 | 修正 |
|---|---|
| 把 `CLAUDE.md` 当产品需求或任务清单 | 把需求放到 spec/issue/docs；`CLAUDE.md` 只写长期执行规则 |
| 把术语、架构、模块说明继续堆在 `CLAUDE.md` 或单个 `project.md` | 将稳定项目知识迁移到根目录 `repository/`，按文件分区维护 |
| 在 monorepo 根目录扫描后写全仓混合说明 | 先确认目标项目；只有明确根级目标才写 monorepo-wide 规则 |
| 根据用户口述添加不存在的命令 | 以脚本和配置为准；冲突时说明“未配置 / 待确认” |
| 复制 README 的人类上手教程 | 摘要成 Claude 需要执行的命令、边界和注意事项 |
| 写“保持代码质量”等口号 | 改成具体规则：运行哪个检查、复用哪个模块、禁止改哪些路径 |
| 在 `repository/` 里记录今日计划、临时决策或对话上下文 | 只保留稳定、可复用的项目知识；临时内容放 issue/spec/任务工具 |
| 记录私有访问地址、账号、token | 删除或泛化；如确需说明流程，写“向用户确认凭据 / 环境” |
| Before Finishing 章节缺失或空泛 | 必须包含可执行检查清单（typecheck、lint、test、git status） |
| Dangerous Operations 章节缺失 | 必须标注高风险操作的 STOP 条件（删除、配置变更、生产部署） |
| 未封堵自我合理化 | 补充“禁止的自我合理化”章节，封堵“改动小不需要测试”等话术 |

## 红旗警告 - 停止并询问或重定向

### 多项目不明确
发现多个应用/包/`package.json`，但用户只说“更新文档”或“初始化项目知识”。停止并询问目标目录；不要因已有根 `CLAUDE.md` 或已有 `repository/` 就猜。

### 用户命令与项目不一致
用户说有 `npm test`、`deploy` 等命令，但脚本不存在。不要写成可用命令；写“未配置”或询问是否要新增脚本。

### 业务需求伪装成长期知识
“把这个 feature spec / 今日计划 / TODO 写进 CLAUDE.md 或 repository 让未来 Claude 记住”。不要直接写入；建议放到 spec/issue/docs，只把稳定架构约定或长期知识写入合适载体。

### 未经验证的常识
“React 项目通常这样”“Node 项目一般这样”。必须找到代码或配置依据，否则不写。

### 继续使用单文件知识承载
已经决定采用根目录 `repository/`，却仍试图用一个 `project.md` 或把百科内容塞进 `CLAUDE.md`。应拆分到知识库目录，并保留清晰职责边界。

### 复制其他文档
README、OpenSpec 或团队规范已有长段说明。只提炼 Claude 工作所需规则与稳定知识，避免双重维护。
