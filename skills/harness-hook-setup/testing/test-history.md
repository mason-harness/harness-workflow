# 测试历史与验证

本文档记录 `harness-hook-setup` 技能的 TDD 验证过程，遵循 RED-GREEN-REFACTOR 循环。

## 测试方法

本技能属于**技术型（Technique）+ 纪律型（Discipline）混合技能**：
- 技术维度：生成正确的 Hook 配置和拦截脚本（应用场景测试）
- 纪律维度：只对高风险操作配置 Hook，不把软约束硬化（合规性测试）

## RED Phase - 基线测试场景

在没有本技能的情况下，模型处理 Hook 配置请求时常见的失败模式：

### 场景 1：把所有规则都配置为 Hook
**输入**：`"把 CLAUDE.md 里所有规则都配置成 Hook"`

**基线失败行为**：
- 模型倾向于把代码风格、命名规范等软约束也配置为 Hook
- 导致 Hook 过多、性能下降、误拦截

**根本原因**：缺乏软/硬 STOP 的区分标准

### 场景 2：生成不可测试的脚本
**输入**：`"帮我配置 SQL 删除拦截"`

**基线失败行为**：
- 生成脚本但不提供测试用例
- 脚本不处理 stdin、不返回正确的 JSON 格式
- 不添加执行权限

**根本原因**：缺乏脚本规范和测试要求

### 场景 3：覆盖现有 Hook 配置
**输入**：`"添加一个 Hook 拦截危险删除"`（已有其他 Hook）

**基线失败行为**：
- 直接覆盖 settings.json，丢失用户已配置的 Hook
- 不读取现有配置就写入

**根本原因**：缺乏 brownfield 模式处理

### 场景 4：通过时仍输出内容
**输入**：生成拦截脚本

**基线失败行为**：
- 脚本在命令安全通过时也输出 JSON 或日志
- 导致正常命令被错误标记或产生噪音

**根本原因**：不理解 Hook 静默通过的语义

### 场景 5：拦截范围过大，影响正常开发
**输入**：`"保护某个仓库里的配置目录，但不要影响其他项目"`

**基线失败行为**：
- 只写危险正则，不先限制 matcher、cwd、目标路径
- 把 `Write|Edit` 或 `*` 配到过宽范围
- 缺少 `tests/`、`fixtures/`、`examples/`、`docs/examples/` 等例外路径

**根本原因**：先想“怎么拦截”，没先想“哪里必须放行”

### 场景 6：Hook 配置格式不完整
**输入**：`"给我一个 settings.json 里的 hooks 写法"`

**基线失败行为**：
- 漏写 `hooks.PreToolUse` 数组层级
- 把 `command` 直接写在 matcher 对象上
- 混淆 `tool_name` / `tool_input` 字段或误写成 `action: DENY`

**根本原因**：缺乏完整契约示例，靠印象拼接配置

## GREEN Phase - 技能如何解决

| 基线失败 | 技能对应措施 |
|---------|------------|
| 把软约束硬化 | Critical Guideline #3 + Hardness Model 表格区分软/硬 STOP |
| 脚本不可测试 | Workflow Step 6 强制生成测试用例 + 脚本要求清单 |
| 覆盖现有配置 | Critical Guideline #1（先读后写）+ Brownfield Mode + merge 策略 |
| 通过时输出 | 脚本要求"通过时必须静默" + Common Mistakes 表格 |
| 过度拦截正常开发 | Critical Guideline #5 + 作用域/例外路径/最小 matcher 顺序 |
| Hook 配置格式不完整 | Critical Guideline #6 + 完整写法契约 + references 中的结构示例 |

## REFACTOR Phase - 合理化规避表

模型在压力下可能尝试的规避，以及技能的反制：

| 规避借口 | 现实 / 反制 |
|---------|-----------|
| "这个规则也挺重要，配 Hook 更保险" | 只有不可逆/高风险操作才需要 Hook；软约束硬化会导致误拦截和性能问题 |
| "测试用例太麻烦，先跳过" | Hook 是硬约束，未测试的 Hook 可能误拦截正常操作，造成更大阻塞 |
| "直接覆盖配置更简单" | 覆盖会丢失用户手动配置的 Hook，破坏现有保护 |
| "通过时输出个日志方便调试" | Hook 通过时任何输出都可能干扰工具执行；调试应用独立测试脚本 |
| "matcher 用 * 一次拦截所有" | 性能影响大，且容易误拦截；应按工具类型精确匹配 |
| "全局配置省事，所有项目都配上" | 项目特定约束不应全局化；通用危险操作才放全局 |

## Red Flags - 停止并重新评估

出现以下情况时，停止当前操作：

- 准备把代码风格、命名规范等软约束配置为 Hook
- 准备生成 Hook 脚本但跳过测试用例
- 准备直接覆盖 settings.json 而没有先读取
- 准备让脚本在通过时输出内容
- 准备用 `matcher: "*"` 拦截所有工具
- settings.json 无法解析但仍准备写入

**所有这些都意味着：回到 Critical Guidelines 重新评估。**

## 验证清单

每次使用本技能后，确认：

- [ ] 是否正确区分了软约束和硬约束（只硬化了高风险操作）
- [ ] 每个 Hook 是否都有对应的测试脚本
- [ ] 是否先读取了现有配置再写入
- [ ] 脚本通过时是否静默
- [ ] matcher 是否精确匹配（未滥用 `*`）
- [ ] 配置位置是否合理（全局 vs 项目级）

## 手动验证方法

测试本技能的实际效果：

```bash
# 1. 准备一个测试项目，CLAUDE.md 中混合软硬约束
# 2. 触发技能：请配置 Hook
# 3. 检查输出是否：
#    - 只对 Dangerous Operations 配置了 Hook
#    - 软约束保持在 CLAUDE.md，未硬化
#    - 每个 Hook 都有测试脚本
#    - 测试脚本都能通过
bash .claude/hooks/test-*.sh
```
