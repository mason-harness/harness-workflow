# 上下文治理实践指南

## 基于 OpenSpec + Claude Code 的会话窗口管理

> **更新日期**：2026-07-23

## 什么是上下文治理

**上下文治理** = 把 LLM 的上下文窗口当作与 tokens、证据并列的**有限可耗尽资源**主动管理，而不是等它撑爆后再被动处理。

> 类比：Hardness 是"做什么 / 不做什么"的纪律，上下文治理是"会话窗口怎么保持可工作"的纪律。前者防止产出漂移，后者防止**承载产出的容器**先失效。

### 设计目标

- 承认上下文窗口有上限，且越满后续任务质量越差（漂移、前后约束丢失、"假完成"概率上升）
- 把"上下文不够"从"事后发现质量下降"前移到"阶段门禁主动减负"
- 让会话状态可沉淀、可恢复、可追溯——窗口可以清，但进度不能丢
- 与 Hardness 同构：用外部记忆固化意图，用主动委派隔离工作，用窗口收敛决定何时重置，用证据决定状态是否已安全落盘

**上下文治理不追求永不清空窗口，而追求"清空前后状态都不丢"。**

## 失效诊断

上下文治理失效时，症状往往先表现为"质量下降"而非"窗口报错"。

| 症状 | 可能原因 | 本仓对应修复手段 |
|---|---|---|
| 前后约束丢失，后期不遵守早期规则 | 状态只存在窗口里，未沉淀外部记忆 | 第 1 层：先落 tasks.md/verify.md 再继续 |
| 探索阶段把主上下文撑爆，后续推理变差 | 未委托 subagent，原文灌回主窗口 | 第 2 层：主动委派 Explore subagent |
| /compact 后丢关键状态 | 压缩前未确认证据已落盘 | 第 3 层：压缩前硬触发"先落外部记忆" |
| 跨 session 打开后不知道上次到哪了 | 未用 slices-track 恢复整体推进 | 第 4 层：Archive 后跑 track 或开新会话 |
| 一个 Change 越做越长、约束越丢越多 | 上下文跨多主题长期维持，该切 Change | 第 4 层：会话切分（指向拆分策略文档） |

## 上下文治理四层模型

### 第 1 层：外部记忆（External Memory）

**适用阶段**：贯穿全程，尤其 Apply / Verify  
**核心动作**：把状态从窗口迁到文件  
**与 Hardness 的关系**：复用 Hardness 的证据系统，不另造格式

本仓的 `openspec/changes/<id>/tasks.md` 的 checkbox 与 `verify.md` 的 Evidence 章节，**本身就是跨会话外部记忆**——它们不依赖窗口存活。格式见 `Hardness-Practice.md` 证据系统章节，本文不重述。

**本层新增的纪律**：当窗口接近饱和时，"先落 tasks.md/verify.md 再继续"是一个**硬触发，而不是可选项**。

判定线：

- 本 Change 未完，但窗口已明显变重（推理开始重复、约束开始丢）→ **停下手头新动作**，先把当前进度写回 tasks.md（已完成哪几个 checkbox、当前在做哪个、blocker 是什么），再决定继续还是压缩。
- verify 过程中产生的命令输出，**当场**记进 verify.md 的 Evidence，而不是攒在窗口里等"最后一起写"。

一句话：**状态在文件里，窗口里只放"当前正在做的那一段"。**

### 第 2 层：主动委派 Subagent（Delegation）——核心章节

**适用阶段**：Explore、Apply 前调研、Verify 预扫描  
**核心动作**：把"读很多、产结论少"的工作移出主窗口  
**与 Hardness 的关系**：这是上下文治理的**主动**手段，必须具体可执行

> **原则**：不要等窗口满了才被动委派。一旦识别到"结论能压成 1–3 句，但过程会往主窗口灌大量原文"，就**主动**派 subagent。

#### (a) 主动委派的触发信号

以下任一信号出现，就主动派 subagent，而不是自己读：

1. **探索/搜索会读取 5+ 文件**，或产生大量输出，只为得到一个结论。
2. **跨多文件对账**：tasks ↔ verify ↔ 实现一致性、依赖关系核对、规则覆盖检查。
3. **长输出只需判定结论**：verify.md / 测试报告很长，但只需要回答"通过没 / 缺哪条证据"。
4. **并行可独立的调研**：多条无依赖的问题，可以一次派多个 subagent 同时查。

判断线（主动派 vs 自己干）：

| 情形 | 主动派 | 自己干 |
|---|---|---|
| 结论能压成 1–3 句，过程会灌大量原文 | ✅ | |
| 需要多个独立调研并行进行 | ✅ | |
| 需要主上下文基于结论做决策 | | ✅（subagent 给依据，主上下文决策） |
| 动作是写文件/改代码/勾选 checkbox | | ✅（这是 apply 的职责，不委派） |

#### (b) 委派契约模板（给 subagent 的输入/输出结构）

一个合格的委派必须**同时**约束输入边界和输出形态，否则 subagent 会把原文回传，反而加剧主上下文负担。

**输入骨架**（写给 subagent 的 prompt 至少包含）：

- **范围**：查哪些文件 / 目录 / 命令，**只读，不得改动**。
- **问题**：要回答什么（一个明确的、可判定的问题，不是"看看这堆代码"）。
- **约束**：用哪些已验证命令（见 `skills/openspec-slices-register/references/cli-contract.md`），禁用未验证能力；不得跨会话保持状态。

**输出契约**（必须强制）：

- **只回传**：结论（1–3 句）+ 证据位置（`文件:行` 或 `命令 + 输出摘要 + 时间戳`）。
- **禁止回传**：整段原文、完整文件内容、大段日志。证据格式复用 Hardness 证据系统的"合格证据最小要素"（命令/输出/时间戳/位置），定义见 `Hardness-Practice.md`，本文不重述。

**模板示例**：

```text
[范围] 只读 openspec/changes/<id>/ 下 tasks.md、verify.md、实现文件，不得改动。
[问题] 逐条核对 tasks.md 每个 checkbox 是否在 verify.md 有对应证据，列出缺失项。
[约束] 只用已验证的只读命令；不得跨会话保持状态。
[输出要求] 回传：① 结论（哪些 checkbox 缺证据 / 全部齐全）；② 证据位置（checkbox 文本 + verify.md 对应章节定位）。禁止回传文件原文。
```

#### (c) 两类委派分工

1. **verify 只读核对**：已在 `Hardness-Practice.md:252–266` 与 `skills/harness-openspec-verify/SKILL.md:110–117` 覆盖（只读、校验并回传结论、不替代 Hook 硬拦截、不替代真实测试执行）。本文**只指向，不重述**。

2. **探索委托**（本仓空白，写全）：

   - **何时用**：探索期、Apply 前方案调研（"现有有没有可复用实现"、"改这里会影响哪些下游"）、Verify 前预扫描（"先扫一遍再正式核对"）。
   - **并行委派**：多个无依赖问题，一次派多个 subagent 同时查（各自独立窗口），主窗口只收各 subagent 的结论 + 证据位置。
   - **收敛**：把多个 subagent 结论汇总成一份"发现清单"再决策——**汇总在主上下文做**，因为"基于发现做决策"是主上下文的职责（见反模式 d）。
   - **边界对齐**：subagent 有独立上下文窗口、不能跨会话持久化（除非把结果写入文件）、子 agent 间不能直接通信只能各自回传——见 `ClaudeCode原语能力边界.md §5`。

#### (d) 委派反模式（MUST NOT）

- ❌ 让 subagent 回传大量原始内容（违背减负初衷，等于把负担搬回主窗口）。
- ❌ 用 subagent 做硬拦截（硬门禁是 Hook 的职责，见 `skills/harness-hook-setup/SKILL.md`）。
- ❌ 依赖 subagent 保持跨会话状态（结果必须落 tasks.md/verify.md 才算持久；subagent 结束即销毁）。
- ❌ 把需要主上下文决策的"选方案"全权委托（subagent 给依据，主上下文做决策——决策不外包）。
- ❌ 委派后不验收证据位置（结论必须有可追溯定位，否则与"无证据宣称完成"同罪）。

### 第 3 层：窗口收敛（/compact 与 /clear 时机）

**适用阶段**：阶段切换、窗口接近饱和  
**核心动作**：压缩或清空窗口，但**前置条件**是状态已落外部记忆  
**与 Hardness 的关系**：压缩/清空前用证据系统做"状态已落盘"的硬门禁

| 手段 | 何时用 | 前置条件（硬触发） |
|---|---|---|
| `/compact` | 会话还需继续，但历史已沉淀可安全压缩 | 关键状态已落 tasks.md/verify.md；blocker 已记录 |
| `/clear` | 一个 Change 已交付并归档，下一任务主题切换 | 该 Change 已 archive；进度已落入 track 或 tasks.md |

**硬规则**：压缩或清空前，必须确认 tasks 与 verify 证据已落盘；缺失则视为未完成，先补证据再压缩。

判定流程：

```text
窗口变重 / 阶段切换
  ├─ 本 Change 未完？
  │   ├─ 是 → 先落 tasks.md 进度 → /compact → 继续
  │   └─ 否（已归档）
  │       ├─ 有下一个 Change？
  │       │   ├─ 是 → 状态已落 track → /clear → 新会话
  │       │   └─ 否 → /clear 释放窗口
  │       └─（归档未完成 → 不许 /clear，先走 verify/archive 门禁）
```

### 第 4 层：会话切分（Session Split）

**适用阶段**：需求过大、跨多主题长期维持上下文  
**核心动作**：把"一个大需求长时间维持上下文"识别为**该切 Change 的信号**，并默认在归档后开新会话

- **切 Change 的信号**：当一个 Change 已经大到需要长时间跨多个主题维持上下文，通常说明应该进一步拆分。拆分方法见 `OpenSpec Change 拆分策略文档.md`，本文不重述。
- **归档后开新会话**：已归档一个 Change 后，主动 `/clear` 开新会话，避免旧上下文（旧规则、旧范围）污染新 Change。这是默认纪律，不是可选优化。
- **跨 session 恢复**：新会话开头，用 `openspec-slices-track` 恢复整体推进上下文（指向 track skill 边界，不重述其流程）。

## 时机判断（与本仓流程对接）

| 阶段 | 上下文治理动作 |
|---|---|
| **Explore** | 探索产出量大→**主动**委托 Explore subagent（按第 2 层 a/b/c 契约），主窗口只收结论+证据位置，不收原文 |
| **Apply** | 一次只推进一个 checkbox（已由 `harness-openspec-apply` 规定，此处只衔接）；窗口接近饱和且本 Change 未完→先落 tasks.md→`/compact`→继续 |
| **Verify** | 长 verify.md 对账→委托 subagent（指向 `harness-openspec-verify` 已有规则）；证据当场落 verify.md |
| **Archive 后** | 状态已归档→`/clear` 开新会话，或跑 `openspec-slices-track` 恢复整体推进 |

### 开发期读写污染与减负（Apply 期展开）

Apply 期是主上下文最易被撑爆的阶段：读制品（proposal/specs/design/tasks）、读源码、写代码、跑命令，每一类都往窗口灌内容。读与写对窗口的污染不对称，减负杠杆也不同。

**读与写的污染不对称**

| 污染源 | 进窗口的内容 | 可减程度 | 主要减负手段 |
|---|---|---|---|
| 探索性读（读一堆文件找方向/对账） | Read 的 `cat -n` 全文，常驻窗口 | ✅ 大幅可减 | 委派 Explore subagent（第 2 层契约），原文进隔离窗口，主上下文只收结论+证据位置 |
| 定向读（为改某段而读源码/制品） | 同上 | ✅ 可压缩 | Read 用 `offset`/`limit` 只读相关行，别整文件读 |
| 写代码（Write） | 整文件内容作为 output token | ⚠️ 部分不可避免 | 用 Edit 代 Write（只发 `old_string`+`new_string`），避免反复重写整文件 |
| 已完成的读/写堆积 | 死重，占窗口 | ✅ 可清 | 子任务做完、状态落 tasks.md/verify.md 后，旧内容是死重→`/compact`（第 3 层） |

**关键不对称：读可委派，写不可外包。** 探索读可整个搬进 subagent 隔离窗口（最大杠杆）；但"改这段代码"需要主上下文决策，不能外包给 subagent，只能用 Edit 代 Write、定向读代整文件读来压体积。

**读制品（proposal/design/tasks）时的减负**

> 注意：制品本身臃肿（tasks 几十项、design 过长）是 task&design 线问题，治本是拆 Change，见 `OpenSpec Change 拆分策略文档.md`，本文不重述。这里只管"制品已存在、必须被 main 读取时怎么读才不污染窗口"。

- tasks.md 只读当前 checkbox 与相邻依赖行，不全读（Read 用 `offset`/`limit`）。
- design.md 先读"推荐方案"段与当前责任单元相关段；替代方案/背景段按需再读，不一次灌入。
- 跨制品对账（tasks↔verify↔实现一致性）走第 2 层委派契约，不让原文回主上下文。

一句话：**读制品要定向或委派，别整份灌进窗口；写代码用 Edit 代 Write；做完的子任务立即落盘，让旧内容变可清的死重。**

## 证据与可追溯

上下文治理动作本身也要留痕，避免"清空了才发现没存"。

- `/compact` 前、`/clear` 前，在 verify.md 或 tasks.md 记一句：状态已落盘 / 已归档 / 下次从哪恢复。
- 一条 Hard Rule（复用 Hardness 简式公式：对象 + 动作 + 失败处理 + 证据）：

> 会话压缩/清空前必须确认 tasks 与 verify 证据已落盘；缺失则视为未完成，先补证据再压缩，证据位置记录在 verify.md。

- 委派 subagent 得出的结论，若影响决策，其证据位置（文件:行 / 命令摘要）必须随结论一起落入主上下文可见的文件，而非只留在 subagent 的销毁窗口里。

---

## 禁忌

1. **不要在探索期过度硬约束**：与 Hardness 禁忌一致，探索期低硬度，允许发散——但探索**产出**要主动委派，不等于探索过程要硬约束。
2. **不要把外部记忆当"写完就忘"**：落盘仍需按证据格式（命令/输出/时间戳），否则等于把"假完成"从窗口搬到了文件。
3. **不要用 subagent 做硬拦截**：已是本仓既有边界，见 `ClaudeCode原语能力边界.md §5.3`。
4. **不要在不落外部记忆的前提下直接 `/clear`**：清空窗口前状态未落盘 = 丢失进度。
5. **不要把决策外包给 subagent**：subagent 给依据，主上下文做决策。
6. **不要让 subagent 回传原文**：违背减负初衷。

## 渐进式采纳

### 阶段 1：先落外部记忆

目标：让"窗口可以清，但进度不丢"先落地。重点用第 1 层——tasks.md/verify.md 当外部记忆，窗口变重先落盘再继续。

### 阶段 2：主动委派

目标：让探索和对账不再撑爆主窗口。重点用第 2 层——按触发信号主动派 subagent，强制输出契约（只回结论+证据位置）。

### 阶段 3：窗口收敛 + 会话切分

目标：建立完整的"压缩/清空/切会话"护栏。重点用第 3、4 层——压缩前硬触发落盘，归档后默认开新会话。

| 团队情况 | 建议起点 |
|---|---|
| 刚开始用本仓流程 | 阶段 1 |
| 流程跑通但会话常撑爆 | 阶段 1 + 阶段 2 |
| 多 Change 并行、跨 session 推进 | 阶段 3 |

## 常见问题

### Q1：上下文治理会降低开发效率吗？

短期会增加"落盘/委派"的动作成本，但能减少窗口撑爆后的返工、约束丢失导致的"假完成"，以及跨 session 找回进度的成本。

### Q2：什么时候必须用 subagent，什么时候自己干？

见第 2 层 (a) 判断线：结论能压成 1–3 句、过程会灌大量原文→主动派；需要主上下文基于结论决策、或动作是写改勾选→自己干。

### Q3：/compact 和 /clear 怎么选？

本 Change 未完、还需继续→`/compact`；已归档、要切主题→`/clear`。两者前置条件一致：状态必须先落盘。

### Q4：委派的 subagent 结论丢了怎么办？

结论必须随证据位置一起落入主上下文可见的文件（tasks.md/verify.md）。subagent 结束即销毁，不能依赖它保持状态——这是 `ClaudeCode原语能力边界.md §5` 的边界。

## 总结

上下文治理的核心不是"永不清空窗口"，而是：

> **用外部记忆固化状态，用主动委派隔离繁重工作，用窗口收敛决定何时重置，用证据决定状态是否已安全落盘。**

一句话概括：

> **状态在文件里，窗口里只放当前在做的那一段；读多产少的活主动派 subagent，清空前先证明状态已落盘。**
