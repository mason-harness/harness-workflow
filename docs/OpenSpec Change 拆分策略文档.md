# OpenSpec Change 拆分策略文档

## 一、文档定位

本文档只讨论 **OpenSpec Change 应该如何拆分**。

重点是回答三个问题：
- 什么时候应该拆分
- 应该按什么维度拆分
- 拆分后如何保持依赖、边界与整体演进清晰

本文档不讨论具体工具、技能、登记流程或执行分工，只提供可直接套用的拆分策略。

## 二、核心目标

Change 拆分的目标不是把工作切得越碎越好，而是让每个 Change 都满足以下要求：
- 有清晰目标
- 能独立推进
- 依赖关系明确
- 范围边界稳定
- 归档后整体系统仍然清晰可演进

拆分得太少，会形成“大泥球”；拆分得太多，会形成“碎片化”。好的拆分应该在两者之间取得平衡。

## 三、核心原则

### 原则1：先判断是否真的需要拆分

不是所有需求都需要拆成多个 Change。

如果一个需求同时满足以下特征，通常可以先作为单个 Change 处理：
- 责任单一
- 影响范围有限
- 改动规模较小
- 不需要跨多个独立交付阶段推进

如果一个需求已经天然包含多个目标、多个阶段或多个明确依赖点，就应考虑拆分。

### 原则2：以“可独立推进”作为首要标准

一个好的 Change，应该能清楚回答：
- 这一片单独在做什么
- 它依赖什么
- 它完成后给后续留下什么

如果拆出来的某一片本身没有清晰目标，只是“为了拆而拆”，那通常不是合理拆分。

### 原则3：优先保证边界清晰，而不是形式整齐

拆分不是为了让目录看起来平均，也不是为了追求固定数量的切片。

比起“每片大小差不多”，更重要的是：
- 每片是否有稳定边界
- 每片是否容易描述范围
- 每片是否容易判断完成状态

### 原则4：拿不准时先合后拆

当需求边界还不清楚、依赖关系仍在变化，或者一时无法确定最合适的拆分方式时，优先先合并规划，再在边界稳定后拆分。

先合后拆通常比过早横切更稳，因为过早拆分容易制造伪边界和循环依赖。

### 原则5：把 AI 可稳定处理的规模也视为拆分信号

在 AI 参与规划、生成和校对的流程里，单次 Change 如果承载的修改内容过大，AI 对范围、依赖、边界和完成状态的把握能力会明显下降。

常见表现包括：
- 上下文持续变长，前后约束容易丢失
- 多个目标混在一起，边界判断开始模糊
- 前序决定被后续修改覆盖，整体一致性下降
- 局部看起来都合理，但放在一起后整体质量下滑

因此，判断是否拆分时，不仅要看“理论上能不能放进一个 Change”，也要看“这个规模是否还能被稳定理解和推进”。

如果一次 Change 已经大到需要 AI 长时间跨多个主题来维持上下文，通常就说明应该进一步拆分。

### 制品臃肿早期信号（定性自检）

原则 5 列的是"AI 把握能力下降"等抽象表现（上下文变长、边界模糊、一致性下降、整体下滑），偏难观察。下面把"该拆"前移到**制品级早期信号**——在 proposal / tasks / design 三个 artifact 上就能直接对照，不必等 AI 表现退化才反应。信号都是定性的，不设量化阈值，避免在缺乏依据时凭空定数字。

| 制品 | 早期信号（定性） | 指向哪条已有原则 / 判定 |
|---|---|---|
| **Proposal** | 一个模糊指令被展开成几十个任务、几百行文档，连人都难以一眼评审与把控 | 原则 5（规模超出 AI 可稳定处理范围）、拆分决策树入口（范围是否清晰→是否单一责任/目标） |
| **Tasks** | tasks.md 冗长，checkbox 之间缺清晰逻辑分组与依赖关系，AI 执行时容易"迷路" | 决策树"是否单一责任/目标"分支、第六节依赖管理（顺序/并行/范围边界）、第九节自检"能否一句话说清目标/依赖/不做什么" |
| **Design** | design.md 试图在一次变更里解决所有问题，方案过于复杂，难以聚焦核心 | 原则 3（优先保边界清晰而非形式整齐）、原则 1（拿不准先合后拆）、第八节误区"为了平均而拆/边界含糊" |

**与原则 5 的关系**：原则 5 信号是"AI 已开始把握不住"（后期、抽象、难观察）；本表是"制品本身已臃肿"（前期、可对照、早一步）。后者是前者的**前驱信号**——制品臃肿若不拆，下一步就是 AI 把握能力下降。

**只做观察前端，治本指向原文**：识别到信号后，按哪条维度拆、依赖怎么管、模式怎么选，仍回到决策树（第四节）、维度（第五节）、依赖管理（第六节）、模式（第七节）、合理判定（第九节），本表不重述这些方法。中途顺手加料的蔓延属"范围边界"问题，见第八节误区"范围不断蔓延"。

## 四、拆分决策树

可按下面的顺序判断是否需要拆分：

```text
收到一个需求
  ↓
范围是否清晰？
  ├─ 否 → 先澄清范围，不急着拆
  └─ 是
       ↓
是否属于单一责任、单一目标？
  ├─ 是
  │    ↓
  │   改动规模是否仍然可控？
  │    ├─ 是 → 可以先做单个 Change
  │    └─ 否 → 继续判断是否应拆分
  │
  └─ 否
       ↓
是否存在多个可独立推进的阶段或价值点？
  ├─ 否 → 先合并处理
  └─ 是
       ↓
这些阶段之间是否存在明确依赖？
  ├─ 否 → 可并行拆分
  └─ 是 → 按依赖顺序拆分
```

这个决策树的核心不是追求“拆”本身，而是避免两种极端：
- 明明应该拆，却塞进一个 Change
- 明明不该拆，却拆出一堆没有独立意义的小片

## 五、按什么维度拆分

### 5.1 单一项目场景：优先按用户价值做垂直切片

在单一项目或单一产品能力里，优先按用户可感知的完整价值路径拆分，而不是过早按技术层拆分。

所谓垂直切片，指的是每一片都尽量覆盖一个完整目标所需的关键路径，而不是只完成 UI、只完成 API、或只完成数据库的一部分。

#### 不推荐的拆法

```text
frontend-ui
backend-api
db-query
error-handling
```

问题在于：
- 前端片可能没有真实可用能力
- 后端片可能暂时无人使用
- 数据层片可能只是孤立准备工作
- 多个切片都不能单独表达一个完整结果

#### 更推荐的拆法

```text
s01-export-foundation
s02-export-progress
s03-export-retry
```

这种拆法的好处是：
- 第一片先建立基础主路径
- 第二片在主路径上增强体验
- 第三片补齐韧性与边缘场景
- 每一片都能描述为“对已有能力的完整增强”

### 5.2 多系统或多职责场景：按边界切分

当需求天然跨多个系统、多个仓库、多个服务，或多个明确职责面时，应优先按这些真实边界拆分。

此时如果仍强行按单一用户故事竖切，往往会把一个 Change 变成多个边界的混合体，导致责任不清、依赖难管。

#### 不推荐的拆法

```text
01-login-story
02-session-story
03-permission-story
```

这种拆法的问题通常是：
- 每片都同时跨多个职责面
- 谁先做、谁后做不够清楚
- 一旦某一边界变化，整个切片都要改

#### 更推荐的拆法

```text
s01-auth-provider
s02-gateway-integration
s03-frontend-login
```

这种拆法的优势是：
- 每片都对应清晰边界
- 依赖链容易表达
- 更适合逐步推进和分别验证

## 六、拆分后的依赖管理

拆分之后，最容易失控的不是“切多少片”，而是“切完以后彼此关系不清楚”。

因此每个 Change 至少应明确三类信息：
- **目标**：这一片到底完成什么
- **依赖**：它依赖哪些前序能力或前提条件
- **边界**：这一片明确不做什么

### 6.1 顺序依赖

如果后一个 Change 必须建立在前一个 Change 的结果之上，就应显式表达先后顺序。

典型例子：
- 先完成基础主路径
- 再补进度反馈
- 最后补错误重试

### 6.2 并行拆分

如果多个 Change 之间没有直接前置依赖，只是共同服务于同一个较大目标，则可以并行推进。

但并行的前提是：
- 彼此边界稳定
- 不会频繁争抢同一块核心结构
- 不会因互相等待导致伪并行

### 6.3 范围边界

每个 Change 都应明确：
- 本次要完成什么
- 本次明确不做什么

“本次不做什么”非常重要，因为很多 Change 失控不是因为起点太大，而是因为中途不断顺手加内容。

## 七、常见拆分模式

### 模式1：基础主路径 → 体验增强 → 韧性补全

适用于单一产品能力的渐进增强。

示例：
1. 先做基础可用版本
2. 再做进度、反馈、取消等体验增强
3. 最后补错误处理、重试、恢复等韧性能力

### 模式2：基础层 → 集成层 → 使用层

适用于天然存在分层依赖的场景。

示例：
1. 先提供底层能力
2. 再做中间集成
3. 最后接入最终入口

### 模式3：先合后拆

适用于边界尚不稳定的早期阶段。

做法是：
1. 先按单个 Change 理解整体需求
2. 在实现或设计逐渐清晰后识别稳定边界
3. 再按清晰边界重新拆出多个 Change

## 八、常见误区

| 误区 | 表现 | 更好的做法 |
|---|---|---|
| 拆分过晚 | 一个 Change 已经同时承载多个目标和大量依赖 | 尽早按阶段或价值点拆开 |
| 拆分过早 | 需求还不稳定就硬切很多片 | 先合并理解，再拆 |
| 按技术层机械拆分 | UI、API、DB 各自成片，但都不是完整结果 | 单一项目优先按用户价值垂直切 |
| 为了平均而拆 | 每片大小差不多，但边界含糊 | 先保边界清晰，再看规模 |
| 依赖不显式 | 后续必须依赖前序，但文档里没说明 | 明确写出前后关系 |
| 范围不断蔓延 | 一边做一边顺手塞更多内容 | 明确写出本次不做什么 |

## 九、如何判断拆分是否合理

可以用下面这组问题做快速自检：

- 每个 Change 是否都有一句话能说清目标？
- 每个 Change 是否都能说明自己依赖什么？
- 每个 Change 是否都能说明自己明确不做什么？
- 拆出来的每一片是否都比“技术准备工作”更接近一个完整结果？
- 整体顺序是否一眼能看懂？
- 如果删掉其中一片，其他片的意义是否仍然清楚？

如果这些问题大多答不上来，通常说明还没有拆到合适粒度。

## 十、总结

好的 OpenSpec Change 拆分，核心不是“切得多细”，而是：
- 该拆时拆
- 不该拆时不硬拆
- 单一项目优先按价值垂直切
- 多边界场景优先按真实边界切
- 始终让依赖、范围和推进顺序保持清晰

一句话总结：

> **用最少但足够清晰的 Change，表达完整而可演进的交付路径。**
